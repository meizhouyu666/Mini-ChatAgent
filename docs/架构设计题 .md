模块一：Context/Performance

2.一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

答：这个问题感觉属于context工程和prompt工程的范畴了，如何确保压缩后的对话仍然流畅，如果要做到响应仍然快速，会不会包括prompt cache的部分？ 我觉得最关键的可能还是要考虑context的抽取和压缩。比如用一个小模型从对话记录中抽取关键信息，偏好以及task state。丢弃可能会造成幻觉，冗余的内部推理过程等等垃圾信息，维持信噪比。替代前面很多轮的臃肿文本。连续聊了200轮，说明context太满，导致现在每轮对话要注入的prompt很长，也就说明上游模型算的更久，这就是我们讨论的影响流畅性的原因。

所以关键是要做到压缩状态，一个好的context必须是结构化的，有对应槽位的，比如说，最近的n轮对话，我们选择保留，更早的对话，我们做摘要保留，结构化保留；稳定事实性信息，属于memory那边拼接过来的，可能包括偏好，identify还有背景信息之类的。经过这样结构化的处理context。最后拼接成结构化的prompt，他就一定包含了让我们这次对话也能够流程的必要条件，比如说前面确认的事实，用户的偏好，约定的表达之类的，减少让llm出现“不知道” 的情况，可能是保证流畅的必要条件。



模块二：Memory

&#x20; 1.和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

答：这里我认为涉及到agent记忆工程本身的设计，当然我认为记忆工程本身的设计很大程度上是要取决于agent的业务场景的，这里我们讨论的是聊天Agent，那此时一个好的记忆工程必须包括以下几个结构： 1 STM 也就是短期记忆 2.preference 偏好记忆 3.LTM 长期记忆 4.也许还需要图增强记忆 以应对多跳搜索  还有一个方面就是记忆的加工并存储 记忆肯定不是聊天对话的原文 而是结构化的事实 摘要。

这里说到 和聊天agent熟悉半个月，问了一个以前问过的问题，大概有两种情况，发生在此对话的不久之前问过的问题 或者是另外的某个对话问过的问题 不管怎样 都会在记忆工程中发起检索，路由分发处理过的query，大概流程是会话记忆->长期记忆->事实校验。 召回的是可复用的结论， 然后再做prompt拼接 完成这次query的回答，还要按模式分流，如果这个以前问过的问题包括tools调用 那么tool的状态也是要考虑的一点，根据记忆类型的不同，才决定要复用到什么程度。 相关性，重要性，时效性，置信度 都是综合排序召回结果的因素。



模块三: Task

1.对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

答：从原因出发，忘掉目标的原因是执行长程任务中，积累的context太多，最初的目标由于llm注意力稀释的问题权重降低，所以表现出遗忘的问题，最普遍的方案是做task拆解，拆解成一个个subtask来分别执行子任务，这样做的目的是每一个subtask都是由一个全新的sub agent来处理，所以context窗口也是全新的，可以有效的解决注意力稀释的问题。但是这样做的目的可能就是每个agent由于失去了对项目整体的了解，也许并不能很好的完成我们整体的目标，多agent协作带来的问题就是复杂的多agent编排的问题，每个subtask也许会有对应的依赖关系，都是我们要考虑的范畴。如果从context治理的角度上出发，我们会考虑对最初的目标做定期的注入，筛选并清理会造成注意力分散的历史上下文做状态收敛，维持窗口的信噪比正常，可能可以减少注意力稀释的问题。



模块四：Tool/Session Runtime

1.Agent 工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

答：这里我认为和微服务架构和分布式系统中的思想很像，事件驱动架构设计，简单举个例子，比如说一些耗时任务，落库刷盘之类的，通过消息队列来做后台异步执行。Agent调用异步工具时，可以先返回状态响应，首先先避免一直无响应死等的状态，等到工具执行完成之后，才真正回调结果。 runtime拦截到工具调用意图的时候。不阻塞主loop，异步脚本干活。 本质上就是让思考和推理的节点和tools调用的节点解耦。整个runtime的推进或者说整个链路的流转依赖状态驱动，状态的改变依赖事件触发（所以工具调用就是事件，开始和结束的状态回调驱动上层推进。而且跟产品开发的思想一样，等待的过程中可以输出中间状态，比如说输出正在处理中。思考中，分析中之类的，在交互层面上也可以缓解用户等待的情绪。而且task的执行也可以通过DAG的方式，并发执行并不依赖这次重tools调用结果的其他节点。核心runtime可以正常的处理这些node。



模块五：

1.Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

答：claude code 的工具输出方式并不是严格的结构化json，基于openai统一协议的都是输出的结构化json，这样做的好处首先肯定是生态好了，首先就兼容llm的灵活切换，而且还与后端生态契合，json都支持反序列化。而且，强结构化就意味着约束能力强，可以避免模型层面的发散。但是缺点同样明显，json只有完整输出 意义才完整，不完整的json没办法解析，这就限制了stream输出的可能性。

关于为什么claude code 要选择xml标签的形式，我是这样想的，这样的设计服务于他最主要的coding能力，xml就是简单的标签形式，在自然语言的文本中可以拦截到这些标签，对比openai的json，就可以做到自然的流式输出，不过xml就没办法强类型校验和约束了。不过authropic他们非常强大的模型能力又弥补了这点，而且authropic他们很多优化都内置在模型层面上了，例如propmt caching这块。还有一点就是xml生态，没办法复用现成框架的工具解析组件。得自己写拦截状态机。

所以 ，总的来说，两种设计可以说是各有优劣的，如何取舍我个人认为还是根据业务场景，claude code就是服务于coding的，从模型到agent设计。这里的xml就是一个例子。直接向终端输出原始的代码，不要还要解析转义的json。

然后国内别的这些基于openai的，感觉就偏向于各种业务场景的灵活迁移运用，又兼容这些很多框架，以便低代码程度的搭建agent。像是一个通用业务agent底座这样的感觉。而且还支持后端生态，更合适了。
